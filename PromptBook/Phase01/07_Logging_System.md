# 1. Overview & Philosophy

The logging system relies exclusively on **structured logging** to ensure that logs are both highly readable for developers on the console and machine-parsable for future analytics and debugging.

**Core Principles:**
- **No string interpolation in log messages.** Variables are passed as kwargs (`logger.info("Scraping started", slug="two-sum")`), ensuring they are queryable keys in the JSON payload.
- **Context is bound, not passed.** Identifiers like `pipeline_id` and `slug` are bound to the logger instance so they automatically append to every log emitted by a module.
- **Dual Output:** Colorful, human-readable output to the terminal, and strict JSON output to rotated log files.

---

# 2. Logging Architecture

We use `structlog` layered over standard Python `logging`.

### 2.1 Processor Pipeline
All logs pass through a unified processor pipeline before being routed to a handler:
1. `TimeStamper`: Adds ISO 8601 UTC timestamps.
2. `add_log_level`: Appends the severity level (INFO, DEBUG, etc.).
3. `add_logger_name`: Appends the module origin.
4. `ContextVars`: Injects contextual data (Pipeline ID, Request ID).
5. `format_exc_info`: Formats exceptions into a structured stack trace string.

### 2.2 Routing
After processing, logs are branched:
- **Console Handler:** Uses `ConsoleRenderer` to render colorful, human-friendly output for the developer.
- **File Handler:** Uses `JSONRenderer` to write strict NDJSON (Newline Delimited JSON) to log files.

# 3. Contextual Data & Identifiers

To trace events across 9 discrete modules, context must be maintained.

### 3.1 Pipeline ID (`pipeline_id`)
A unique UUID (e.g., `run_8f3a2b1c`) generated by the Orchestrator at the start of a batch run. All logs across all problems in that run share this ID.

### 3.2 Problem Slug (`slug`)
When the Orchestrator begins processing a specific problem, it binds the `slug` (e.g., `two-sum`) to the logger passed to the modules.

### 3.3 Request ID (`request_id`)
For external API calls (LeetCode, Gemini, YouTube), a random `request_id` is generated immediately before the HTTP call. If the request fails and retries, the `request_id` allows tracing the specific lifecycle of that attempt.

# 4. Log Levels & Naming

| Level | When to use |
|---|---|
| **DEBUG** | I/O details, prompt payloads, raw API responses, FFmpeg commands. Hidden by default. |
| **INFO** | Progress markers, module transitions, execution times, success events. |
| **WARNING** | Retries triggered, API rate limits hit, graceful degradation events (`NonCriticalError`). |
| **ERROR** | `CriticalError` failures that halt a specific problem but allow the pipeline to continue. |
| **CRITICAL** | System panics, OOM kills, unhandled OS signals that crash the entire Orchestrator. |

### Module Logger Naming
Loggers are named identically to their module packages:
- `pipeline.orchestrator`
- `pipeline.scraper`
- `pipeline.animation`

# 5. Console vs. File Logging

### 5.1 Console (Developer Mode)
- **Format:** Plain text, aligned, colorized.
- **Default Level:** `INFO`
- **Debug Mode:** If `--debug` is passed or `PIPELINE_ENV=development`, the console level drops to `DEBUG`.

### 5.2 File (Production/Audit Mode)
- **Format:** Pure JSONL (JSON Lines).
- **Default Level:** `DEBUG` (always captures everything for post-mortem analysis).
- **Path:** `data/logs/pipeline.log`.

# 6. Performance & Progress Logging

### 6.1 Performance Timing
Use context managers to wrap and log execution time automatically. Do not manually calculate `t1 - t0`.
*Example output field:* `duration_ms=4521`

### 6.2 Progress Logging
Modules must log their entrance and exit. For long-running modules (Manim, Voice), log discrete milestones (e.g., "Rendering section 3/10"). This prevents the pipeline from appearing hung.

---

# 7. Exception Logging

When catching an exception, use `logger.exception()` or pass `exc_info=True`.
The structured logger will capture:
- `exception_class`: e.g., `RateLimitError`
- `error_id`: e.g., `ERR-SCR-001`
- `traceback`: The formatted stack trace.

**Rule:** Never embed exception messages in the log event string.
*Bad:* `logger.error(f"Failed to scrape: {e}")`
*Good:* `logger.error("Scraping failed", error_id="ERR-SCR-001", exc_info=e)`

---

# 8. Rotation & Retention

To prevent disk exhaustion on long-running local setups:
- **Rotation:** Rotate daily at midnight (`TimedRotatingFileHandler`) or when the file hits 50MB.
- **Retention:** Keep the last 14 days of logs. Older logs are automatically deleted by the logging daemon.
- **Archives:** Rotated logs are compressed into `.gz` format to save space.

# 9. Examples

### 9.1 Standard Info Log
**Code:**
```python
self.logger.info("Video script generated successfully", sections=10, tokens=4521)
```
**JSON File Output:**
```json
{"event": "Video script generated successfully", "level": "info", "sections": 10, "tokens": 4521, "pipeline_id": "run_123", "slug": "two-sum", "logger": "pipeline.script", "timestamp": "2026-07-22T10:00:00Z"}
```
**Console Output:**
```text
[2026-07-22 10:00:00] [INFO ] Video script generated successfully [pipeline.script] sections=10 slug=two-sum tokens=4521
```

### 9.2 API Retry Warning
**Code:**
```python
self.logger.warning(
    "API rate limit hit, retrying",
    attempt=2,
    max_retries=3,
    delay_seconds=4.0,
    request_id="req_88f2"
)
```

### 9.3 Exception Log
**Code:**
```python
try:
    manim.render(scene)
except Exception as e:
    self.logger.error("Manim subprocess crashed", error_id="ERR-ANI-002", exc_info=e)
```
**JSON File Output:**
```json
{"event": "Manim subprocess crashed", "level": "error", "error_id": "ERR-ANI-002", "exception_class": "manim.SceneError", "traceback": "Traceback (most recent call last)...", "slug": "two-sum", "logger": "pipeline.animation", "timestamp": "2026-07-22T10:05:00Z"}
```
