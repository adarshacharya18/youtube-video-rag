# 1. Executive Summary

This document specifies the Multi-Tiered Caching Strategy for the RAG engine. Because the pipeline relies heavily on third-party APIs (Gemini Embeddings, LLM Query Rewriting, Script Generation), aggressive caching is mandatory. Proper caching reduces API costs by up to 80%, cuts retrieval latency from seconds to milliseconds, and provides extreme resilience against transient network failures during long batch runs.

---

# 2. Multi-Tiered Caching Architecture

### 2.1 Embedding Cache (Ingestion)
- **Target:** The exact float arrays generated by the embedding model for a specific string of text.
- **Mechanism:** LlamaIndex's `IngestionCache`. Before an API call is made, the raw text chunk is hashed (SHA-256). If this hash exists in the cache, the API is skipped, and the cached 768D vector is returned immediately.
- **Benefit:** Allows rapid re-indexing of documents where only a few paragraphs have changed, without paying to re-embed the unchanged paragraphs.

### 2.2 Query Cache (Retrieval)
- **Target:** The rewritten queries generated by the LLM (Module 3, Section 3).
- **Mechanism:** A simple Key-Value lookup mapping the `(ScrapedProblem.slug, tags)` to the array of expanded search queries (e.g., `two-sum` -> `["Hash map complement search theory", ...]`).
- **Benefit:** If the pipeline fails during the Video Assembly module and restarts the next day, the Orchestrator doesn't need to ask the LLM to rewrite the search queries for `two-sum` again.

### 2.3 Search Results & Retrieved Context Cache
- **Target:** The exact `RAGContext` dataclass (containing the Top K filtered, reranked, and ordered chunks).
- **Mechanism:** Once the Context Builder finalizes the payload, the JSON representation of the `RAGContext` is serialized and cached using the `ScrapedProblem.slug` as the primary key.
- **Benefit:** Short-circuits the entire Retrieval Pipeline (Query, Vector Search, BM25, Reranking, Building). A cache hit here drops retrieval time from ~2,500ms to ~5ms.

### 2.4 Prompt Assembly Cache
- **Target:** The final constructed prompt string (Jinja2 template output) injected into the Script Generator.
- **Mechanism:** Hashes the combined state of `(Problem + Context + Config)`.
- **Benefit:** Primarily used for developer debugging and A/B testing prompts without needing to rebuild the entire pipeline state.

# 3. Cache Compression

Raw text and float arrays consume significant disk I/O.
- **Vector Compression:** Embedded floats in the cache are stored in a binary format (e.g., `numpy` `.npy` blobs or msgpack) rather than verbose JSON arrays.
- **Text Compression:** The cached `RAGContext` payloads are compressed using `gzip` or `zstd` before writing to the SQLite cache, reducing the payload size by up to 70% and accelerating read/write speeds on HDDs/SSDs.

# 4. Lifecycle: Expiration & Invalidation

Unlike web servers with Time-To-Live (TTL) expirations, a batch pipeline requires deterministic cache invalidation based on state changes.

### 4.1 Expiration (TTL)
- **No TTL on Embeddings:** An embedded chunk of C++ code never semantically "expires". It is valid forever.
- **Short TTL on Prompts:** Cached constructed prompts expire after 7 days to ensure developers aren't accidentally testing against stale Jinja2 templates.

### 4.2 Strict Invalidation Rules
Caches must be aggressively purged to prevent the pipeline from generating videos based on outdated theory.
1. **Knowledge Base Update:** If the Ingestion Engine detects *any* change to the `data/knowledge_base/` (e.g., a new markdown file is added), the **Search Results & Context Cache** is globally invalidated. *Why?* Because a query for `dijkstra` might now have a better chunk available that it missed previously.
2. **Model Swap:** If the `embedding_model` string in the configuration changes (e.g., `gemini-004` -> `gemini-005`), the **Embedding Cache** is globally flushed.
3. **Template Edit:** Modifying the Jinja2 script template instantly invalidates the **Prompt Assembly Cache**.

# 5. Storage Mechanisms

All caches are strictly local and filesystem-bound, adhering to the project's offline-first mandate.

- **SQLite Key-Value Store:** Standardized backend for the Query Cache and Context Cache. It provides ACID guarantees, preventing cache corruption if the pipeline is force-killed (SIGINT) mid-write.
- **Location:** `data/cache/rag/`
- **Isolation:** Each cache tier gets its own `.sqlite` file (e.g., `embeddings.sqlite`, `rag_context.sqlite`) so that blowing away the Context cache doesn't accidentally destroy 50,000 cached embedding vectors.

# 6. Monitoring & Performance

### 6.1 Telemetry Metrics
Every module must log its cache interactions to `structlog`:
- `cache_hit`: Boolean.
- `cache_tier`: e.g., "RAGContext".
- `time_saved_ms`: Calculated delta vs a cold run.
- `bytes_read`: Size of the uncompressed payload.

### 6.2 Performance Targets
- **Hit Rate:** Target >95% hit rate on the Embedding Cache during incremental ingestion.
- **Latency:** A full retrieval cache hit (`RAGContext` tier) must resolve in `< 10ms`.
- **Cost Savings:** The pipeline Orchestrator aggregates the `cache_hit` metrics at the end of a batch run and prints a CLI report: `Estimated API Savings via Cache: $4.25`. This quantifies the architectural value of the system to the developers.
