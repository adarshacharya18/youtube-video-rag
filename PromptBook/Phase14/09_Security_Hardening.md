# 1. Executive Summary

An automated system running 12-hour video pipelines utilizing powerful external API keys (OpenAI, YouTube) and executing dynamically generated Python code (Manim) poses significant security risks if compromised. This document evaluates the current security posture and establishes the final hardening requirements for the Phase 14 Production Environment.

---

# 2. Security Posture Evaluation

### 1. Secrets Management & Credential Storage
- **Current State:** API Keys are ingested via environment variables (`os.environ`) and encapsulated within `PipelineConfig`. The `_secrets` dictionary utilizes `repr=False` to prevent accidental logging.
- **Risk:** Reading directly from `.env` files on disk in a production VM leaves credentials exposed to arbitrary file read vulnerabilities.
- **Recommendation:** Integrate with a managed secrets vault (e.g., AWS Secrets Manager, HashiCorp Vault) or utilize Kubernetes Secrets during the container orchestration phase.

### 2. Plugin Isolation & Manim Execution
- **Current State:** Manim executes Python code generated by the LLM Planner.
- **Risk:** **CRITICAL**. An LLM hallucination or a prompt injection attack from a scraped LeetCode problem could cause the LLM to output malicious Python code (e.g., `import os; os.system('rm -rf /')`), which the Manim renderer would blindly execute.
- **Recommendation:** Implement strict **Subprocess Sandboxing**. The Manim renderer must execute inside an unprivileged Docker container or a highly restricted `chroot` jail with zero network access and read-only filesystem mounts (except for the designated output `/tmp/manim` folder).

### 3. File Permissions
- **Current State:** Artifacts and SQLite ledgers are currently staged in `/tmp/` during development.
- **Risk:** `/tmp/` is world-readable/writable in many Linux distributions.
- **Recommendation:** Enforce strict POSIX permissions (`chmod 700`) on the persistent storage directory (e.g., `/var/lib/dsa_pipeline/`). The Pipeline Orchestrator must run as a dedicated, non-root system user (`dsa_worker`).

### 4. Input Validation
- **Current State:** Scraped problem descriptions are passed directly to the RAG engine.
- **Risk:** Malicious HTML/JS payloads from scraped websites could poison the RAG context or cause cross-site scripting if viewed in an admin dashboard.
- **Recommendation:** Implement aggressive sanitization (e.g., stripping all HTML tags, enforcing ASCII/UTF-8 limits) before writing scraped data into the ChromaDB vector store.

### 5. Audit Logging
- **Current State:** Structured JSON logging is implemented with Correlation IDs.
- **Risk:** Logs do not specifically tag security-relevant events (e.g., authentication to YouTube, Manim sandbox violations).
- **Recommendation:** Introduce a dedicated `SECURITY` log level or a specific `"event_type": "security_audit"` flag in the JSON formatter to alert DevOps immediately if the sandbox is breached.

# 3. Implementation Guidance

### Manim Sandbox Policy (Docker Profile)
When deploying the system, utilize Docker's security options to explicitly restrict the rendering subsystem:
```bash
docker run --rm \
  --network none \
  --read-only \
  --user 1001:1001 \
  --cap-drop ALL \
  -v /var/lib/dsa_pipeline/cache:/output:rw \
  dsa_manim_renderer \
  manim -qm scene.py
```

### Secrets Injection (AWS Parameter Store Example)
Do not store `.env` files on production disk. Retrieve them in memory at startup:
```python
import boto3
import os

def fetch_production_secrets():
    client = boto3.client('ssm', region_name='us-east-1')
    response = client.get_parameter(
        Name='/dsa_pipeline/prod/OPENAI_API_KEY', 
        WithDecryption=True
    )
    os.environ['OPENAI_API_KEY'] = response['Parameter']['Value']
```

### Dedicated System User
Create a hardened user for the application on the host machine:
```bash
# Create user with no shell access
sudo useradd -m -s /sbin/nologin dsa_worker

# Assign ownership of the production database and cache
sudo chown -R dsa_worker:dsa_worker /var/lib/dsa_pipeline

# Strip access from all other users
sudo chmod 700 /var/lib/dsa_pipeline
```
